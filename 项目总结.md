## 三、遇到的坑（完整记录）

### 坑0：数据源完整性验证缺失 ⚠️⚠️⚠️

**现象**：
- 用户给了证监会官网列表页链接作为数据来源
- 该列表页数据严重不完整：
  - 2024年几乎全缺（只有1期，实际39期）
  - 2023年只有上半年
  - 2025年也不齐全
- 用户自己都没发现这个问题
- 我直接开始采集，没有先验证数据完整性

**原因**：
- 盲目信任用户提供的链接，没有主动检查
- 缺乏"先验证再执行"的意识
- 没有对比官网其他入口（如搜索功能）的数据量

**正确做法**：
1. 拿到数据源链接后，先分析数据完整性
2. 检查：时间跨度是否连续？数量是否合理？
3. 发现缺漏时，主动告知用户并寻找补充方案
4. 在官网内部寻找完整数据获取方式（如搜索功能）

**解决**：
- 使用证监会官网搜索功能，搜索"境外发行上市备案补充材料要求"，标题匹配
- 官网搜索结果才是完整、权威的数据来源

**教训**：
> ⚠️⚠️⚠️ **拿到数据源后，必须先验证完整性！不要盲目信任用户给的链接！**
> 即使是官方来源，也可能存在数据不完整的情况（列表页vs搜索页）。应该主动检查、主动告知用户，而不是直接开始执行。

**更深层的原则**：
> 💡 **用户提出的需求路径不一定准确，甚至很多时候都不准确。**
> 遇到疑似问题时，不要直接执行，而是先询问、一起协作确认。这才是真正的"助手"角色——帮助用户发现问题、修正路径，而不是"听话的工具"。

### 坑1：数据采集不完整 ⚠️⚠️⚠️

**现象**：
- 列表页采集只有部分数据
- 2024年只采集了1期，实际有39期

**原因**：
- 证监会列表页本身数据不完整
- 只采集了列表页显示的内容

**解决**：
- 必须使用官网搜索功能采集
- 搜索关键词："境外发行上市备案补充材料要求"
- 选择"标题匹配"
- 人工核对搜索结果数量

**教训**：
> ⚠️ **采集后必须验证数量是否与官网一致！**

### 坑2：软换行导致内容截断 ⚠️⚠️

**现象**：
- 句子中间被截断
- "请说明卢旻将董事提名权授予ZA"后面没了

**原因**：
- Word 软换行（Shift+Enter）被忽略
- docx转md时没有正确处理

**解决**：
- 识别 `<w:br/>` 标签
- 转换为实际换行

**教训**：
> ⚠️ **docx转md必须处理软换行！抽样对比原文验证！**

---

### 坑3：引导语变体识别不全 ⚠️⚠️⚠️

**现象**：
- 68个文件解析失败（actual_count=0）
- 网站只显示标题没有内容

**原因**：
- 只根据部分样本建立模式库
- 引导语有50多种变体没覆盖

**解决**：
- **全量分析**所有引导语变体
- 建立完整模式库（正则组合覆盖）
- 验证100%覆盖率

**教训**：
> ⚠️⚠️⚠️ **先全量分析再解析！不要边解析边打补丁！**

这是最深刻的教训，用户明确表达了"很失望"。

### 坑4：公司名与问题序号混淆 ⚠️

**现象**：
- "一、阿诺医药"被识别为公司名
- 实际"阿诺医药"是公司，"一、"是下一个问题的序号

**原因**：
- 公司名和问题序号格式相同
- 解析逻辑没有区分

**解决**：
- 识别序号前缀
- 区分公司名和问题序号

---

### 坑5：问题格式识别不全 ⚠️

**现象**：
- 部分公司只有名称没有问题
- 百利天恒、澜起科技等

**原因**：
- 问题格式多样："一、"、"一是、"1."
- 还有无序号格式："你公司..."
- 还有陈述句格式（以句号结尾）

**解决**：
- 扩展问题识别函数
- 支持所有格式变体

---

### 坑6：VitePress SSR 构建失败 ⚠️⚠️

**现象**：
```
ReferenceError: document is not defined
```

**原因**：
- 图表代码在服务器端渲染时访问浏览器对象
- `document`、`window` 只在浏览器存在

**解决**：
- 使用 `<ClientOnly>` 包裹
- 使用 `onMounted` 钩子
- 检测 `window.echarts` 存在后初始化

---

### 坑7：重复 script 标签错误 ⚠️

**现象**：
```
Single file component can contain only one <script> element
```

**原因**：
- ECharts 引入和业务代码分开两个 script 标签
- Vue 组件只能有一个 script

**解决**：
- ECharts 通过 config.mts 全局引入
- 业务代码只用一个 script 标签

---

### 坑8：GitHub Actions 与 Cloudflare 混淆 ⚠️⚠️

**现象**：
- GitHub Actions 显示失败
- 但网站实际已更新

**原因**：
- GitHub Actions 和 Cloudflare Git 是两套独立部署机制
- Cloudflare 通过直接关联 Git 仓库自动部署

**解决**：
- 删除 GitHub Actions workflow
- 只用 Cloudflare Git 集成

**教训**：
> ⚠️ **理解部署机制，避免混淆！**

---

### 坑9：Cloudflare 部署没触发 ⚠️

## 四、核心经验总结

### 1. 数据完整性第一

```
采集后验证数量 → 解析后验证完整性 → 抽样对比原文
```

**必须做的检查**：
- [ ] 采集数量是否与官网一致
- [ ] expected_count == actual_count
- [ ] 不存在 empty_content
- [ ] 抽样对比原文确认无遗漏

### 2. 先分析再解析

**错误做法**：
```
采集 → 解析 → 发现问题 → 打补丁 → 发现新问题 → 再打补丁...
```

**正确做法**：
```
采集 → 全量模式分析 → 建立完整模式库 → 解析 → 验证
```

### 3. 验证必须阻断

```python
# 如果某期解析结果为空但预期不为0，抛出异常
if expected_count > 0 and actual_count == 0:
    raise ValueError(f"解析失败：预期{expected_count}家，实际0家")

# 不能让空数据进入下游！
```

### 4. 理解部署机制

- GitHub Actions：通过 workflow 文件触发
- Cloudflare Git：直接关联仓库，推送自动部署
- **两套独立机制，选一套即可**

### 5. VitePress 开发注意事项

- SSR 环境下不能访问浏览器对象
- 客户端代码用 `<ClientOnly>` 包裹
- Vue 组件只能有一个 `<script>` 标签
